# Phase 5 — Retrieval

## Summary
Takes Phase 4's query variants (raw query + optional HyDE doc + multi-query paraphrases + decomposed sub-questions) and turns them into a final ranked candidate set: dense ANN search + sparse BM25 search per variant, RRF fusion across all of them, then BGE cross-encoder reranking against the original raw query. `RetrievalPipeline.retrieve()` is the single entry point Phase 6 calls. No state left on disk — reads the Phase 3 Qdrant collection, returns an in-memory list.

## Key Design Decisions
- **Query variants are passed into `RetrievalPipeline`, not generated by it**: Phase 4 already owns HyDE/multi-query/decomposition and each needs an LLM call — keeping retrieval LLM-free means it stays testable without an Ollama client, same separation `IndexManager` keeps from chunking.
- **`RetrievedChunk.score` is a single float, not per-source**: dense cosine similarity and BM25 scores are on incomparable scales — only rank within each source's list matters, which is what RRF consumes. The field holds whichever stage most recently set it (raw score → fused RRF score → cross-encoder score), so callers never need to know which stage produced "the current ranking score."
- **RRF (parameter-free) over weighted-sum fusion**: no tunable weight to overfit to a small eval set, and it's the standard hybrid-search baseline.
- **Reranking always scores against the raw query, never a HyDE doc or paraphrase**: those exist only to widen retrieval recall — the user's actual question is what should determine final relevance ranking.
- **`build_query_variants()` dedupes while preserving order**: `DecomposedQuery.sub_questions` is `[raw_query]` when decomposition is off, so without dedup the raw query would double-count in RRF fusion.

## Execution Flow
1. `scripts/retrieve.py --query "..."` invoked
2. `load_config()` resolves `configs/base.yaml`
3. `RetrievalPipeline(cfg)` opens the embedded `QdrantClient`, builds `HybridRetriever` + `Reranker`
4. `retrieve(raw_query, transformed, decomposed)` → `build_query_variants()` collects every distinct query string
5. `HybridRetriever.retrieve(variants)` → dense + sparse search per variant → `rrf_fuse()`
6. `Reranker.rerank(raw_query, fused)` → cross-encoder re-scores, truncates to `top_k_rerank`
7. CLI prints each stage (dense/sparse/hybrid/reranked) as a rich table plus a dense-vs-sparse overlap summary

## Data Contract (Input → Output)
- Input: raw query `str` + optional `TransformedQuery` / `DecomposedQuery` (Phase 4)
- Output (consumed by Phase 6): `list[RetrievedChunk]`, length ≤ `top_k_rerank`, cross-encoder-scored, best-first
